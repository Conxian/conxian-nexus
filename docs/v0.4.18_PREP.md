# Conxian Nexus v0.4.18 Preparation: Resilience & Auth Hardening

This document outlines the implementation path for Hole 3.1 (SRL-1 Recovery) and NIP-006 (JWT/RBAC).

## 1. SRL-1 Recovery Triggers (Hole 3.1)

### Context
Current `LightningResilienceAdapter` in `src/executor/lightning.rs` correctly classifies failures (Permanent, Transient, Indeterminate, MppPartial) but does not yet automate recovery.

### Proposed Triggers
1. **Transient -> Retry**: If a failure is marked as `Transient`, initiate up to 3 retries with exponential backoff before escalating to `Failed`.
2. **MppPartial -> Split Recovery**: If a multi-path payment partially succeeds, trigger a "Residual Re-route" for the remaining amount.
3. **Indeterminate -> Reconciliation**: Payments in `Indeterminate` state must trigger a lookup against the Lightning node after 60 seconds to confirm if they were settled or timed out.

### Implementation Path
- Add `recovery_loop` to `NexusExecutor` or a dedicated `LightningOrchestrator`.
- Use `sqlx` to poll `payment_intents` table for `Recovering` or `Pending` items older than X.

## 2. Admin Token Hardening (NIP-006)

### Context
Current `src/api/admin.rs` relies on a static `NEXUS_ADMIN_API_TOKEN` provided via environment variables.

### Proposed Schema (JWT/RBAC)
- **Token Class**: Use HS256 JWTs signed with a rotated `JWT_SECRET`.
- **Claims**:
  - `sub`: Principal (Admin/Auditor/Operator)
  - `exp`: Expiry (Default 1 hour for write actions)
  - `scopes`: [`nexus:admin:write`, `nexus:admin:read`, `nexus:governance:approve`]
- **Identity Provider**: Initial implementation will allow Nexus to issue its own sessions after valid Dual-Signature login (`POST /v1/admin/login`).

### Implementation Path
1. Define `AdminClaims` struct.
2. Implement `AuthLayer` middleware for `admin_routes`.
3. Add session revocation list in Redis for force-logout capability.
